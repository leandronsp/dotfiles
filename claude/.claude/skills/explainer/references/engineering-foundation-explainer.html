  • ✓Major new feature or crosses domain boundaries

When a droid catches a real class of bug, ask: can this be a linter rule? If yes — add it to .golangci.yml or eslint.config.mjs, remove it from the droid prompt. Over time, droids shrink and sharpen. The deterministic floor rises.

The honest gaps

Why the gaps exist

The

Several key features were built ad-hoc or with vanilla OpenSpec 0.x — no DDD, no custom workflows, no decision log. Drifts and inconsistencies were deferred, waiting for the workflow to stabilize. Now we have OpenSpec with DDD and custom workflows, a decision log for ad-hoc prompts, linear walkthroughs, DESIGN.md as a structured agent-readable design spec, NuxtUI v4 as the accessibility foundation, and a much higher quality gate. The Bug Bash closes what was left behind.

Retrofitting our company’s constraints

Peaky has been learning platform limits in real time. Not being able to call Applications or Products APIs directly forced scope trims mid-build — and those trims left inconsistencies across the codebase. Some are fixable now. Others need the broader team. All are tracked.

make mr — the quality gate

One command, every gate in sequence — run it before opening any MR.

What make mr runs — nine steps in sequence
01 lint oxlint → ESLint + security plugin also in CI
02 lint-docs rumdl on every markdown file also in CI
03 typecheck Nuxt TypeScript strict mode
04 audit pnpm audit --prod only
05 test Vitest unit ~16s also in CI
06 go fmt + vet format + correctness
07 golangci-lint 20 linters, all Go packages
08 test:go unit + in-memory integration
09 test:integration Nuxt integration ~25s

Every step runs locally before an MR is opened. Green-flagged steps also run in CI today — wiring up the rest is tracked in #39.
